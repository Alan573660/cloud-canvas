## 1. Проблема и цель

### Проблема

У каждого поставщика свой формат прайса. Один и тот же товар записан тремя способами:

```
Поставщик А: "Профнастил С-20 0,5мм Полиэстер RAL3005 100×1150мм — 650 руб/м²"
Поставщик Б: "МП-20-R-0.5-PE-3005 — 650"
Поставщик В: "С20 полиэстер красное вино 0,5 — 650р"
```

**Невозможно** сравнить цены, построить каталог, дать ответ клиенту — пока данные не приведены к единому виду.

### Цель

Превратить **любой** входящий прайс в структурированный каталог:

```json
{
  "profile": "С-20",
  "thickness_mm": 0.5,
  "coating": "Полиэстер",
  "color_code": "RAL3005",
  "width_work_mm": 1100,
  "width_full_mm": 1150,
  "price": 650.0,
  "unit": "m2",
  "sheet_kind": "PROFNASTIL"
}
```

### Требования к результату

| Метрика | Цель |
|---------|------|
| Автоматическое распознавание | ≥ 90% строк без вмешательства |
| Время анализа 70k строк | ≤ 45 сек |
| Время полного применения | ≤ 2 мин (async) |
| Повторный импорт того же поставщика | ≥ 97% автоматически (правила запомнены) |
| Количество вопросов пользователю | ≤ 10 на импорт (паттерны, не строки) |

## 2. Общая архитектура

### Кто за что отвечает

```
┌─────────────────────────────────────────────────────────────────┐
│ FRONTEND (Lovable / React)                                      │
│ • Загрузка файла                                                │
│ • Маппинг колонок                                               │
│ • Визуализация кластерного дерева                               │
│ • AI-чат для корректировок                                      │
│ • Подтверждение / отклонение предложений ИИ                     │
│ • НЕ содержит бизнес-логики нормализации                        │
└─────────────────────────────────────────────────────────────────┘
         │                                    │
    Supabase Edge Functions              Supabase DB
    (прокси, без логики)             (правила, состояние)
         │                                    │
         ▼                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│ CLOUD RUN — 3 сервиса                                           │
│                                                                  │
│ 1. price-import-worker                                           │
│    • Парсинг файлов (CSV/XLSX/PDF)                              │
│    • Определение и маппинг колонок                               │
│    • Запись сырых данных в BigQuery                              │
│                                                                  │
│ 2. catalog-enricher ◀── ОСНОВНОЙ SCOPE ЭТОГО ТЗ                │
│    • Классификация товаров по категориям                         │
│    • Извлечение атрибутов (профиль, толщина, покрытие, цвет)    │
│    • AI-обогащение через Vertex AI Gemini                       │
│    • AI-чат для интерактивных корректировок                      │
│    • Запись обогащённых данных обратно в BigQuery                │
│                                                                  │
│ 3. pricing-api-saas                                              │
│    • Чтение готового каталога из BigQuery                        │
│    • Расчёт цен с учётом скидок из Supabase                    │
│    • API для фронтенда и AI-агентов (Dialogflow CX)             │
└─────────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│ BIGQUERY — единственное хранилище товарных данных               │
│ • master_products_current — все товары всех организаций          │
│ • dim_profiles — справочник профилей и ширин                    │
│ • dim_ral_colors — справочник RAL/RR                            │
└─────────────────────────────────────────────────────────────────┘
```

### Принцип разделения

| Слой | Делает | НЕ делает |
|------|--------|-----------|
| **Frontend** | Показывает данные, принимает решения пользователя | Не парсит файлы, не нормализует, не пишет в BQ |
| **Edge Functions** | Проксирует запросы, проверяет auth | Не содержит бизнес-логику |
| **catalog-enricher** | ВСЯ нормализация, AI, запись в BQ | Не хранит состояние (stateless) |
| **Supabase** | Хранит правила, статусы, пользовательские решения | Не хранит товарные данные (они в BQ) |
| **BigQuery** | Хранит все товары, отдаёт через pricing-api-saas | Не обрабатывает, только хранит |

## 3. Жизненный цикл прайса

### Полный путь: файл → каталог

```
Шаг 1: ЗАГРУЗКА
───────────────
Пользователь → выбирает файл → загружает в Supabase Storage
UI создаёт запись import_jobs (status=QUEUED)

Шаг 2: ПАРСИНГ (price-import-worker)
─────────────────────────────────────
Worker получает файл → парсит CSV/XLSX/PDF
→ определяет колонки (id, title, price, unit)
→ если не может → возвращает detected_columns → пользователь маппит
→ пишет сырые строки в BigQuery (master_products_current)
→ обновляет import_jobs status=COMPLETED

Шаг 3: НОРМАЛИЗАЦИЯ (catalog-enricher) ◀── ЭТО ТЗ
───────────────────────────────────────────────────
3a. dry_run: анализ без изменений
   → классификация всех строк
   → извлечение атрибутов regex-пайплайном
   → AI-дополнение для неоднозначных строк
   → формирование вопросов пользователю
   → возврат patches_sample + questions

3b. Пользователь проверяет в UI:
   → смотрит кластерное дерево
   → отвечает на вопросы (паттернами)
   → использует AI-чат для корректировок
   → решения сохраняются в Supabase (bot_settings.settings_json.pricing)

3c. apply_start: применение
   → async обновление всех строк в BigQuery
   → используются сохранённые правила из Supabase
   → polling до DONE

Шаг 4: КАТАЛОГ ГОТОВ
────────────────────
pricing-api-saas читает обогащённые данные из BigQuery
→ /api/catalog/items — каталог для UI
→ /api/catalog/facets — фильтры (профили, покрытия, толщины)
→ /api/pricing/quote — расчёт цены
→ AI-агенты используют каталог для ответов клиентам
```

## 4. BigQuery: структура данных

### 4.1 Таблица master_products_current

Это **единственная** таблица с товарами. Кластеризация по `organization_id`.

```sql
CREATE TABLE master_products_current (
  -- Идентификация
  bq_id           STRING NOT NULL,            -- PK: {org_id}_{sku}
  organization_id STRING NOT NULL,            -- UUID организации
  sku             STRING,                     -- Артикул из прайса
  
  -- Из исходного прайса (заполняет price-import-worker)
  title           STRING,                     -- Оригинальное название
  cat_name        STRING,                     -- Категория из прайса (если есть)
  cat_tree        STRING,                     -- Полный путь категории
  price_rub_m2    FLOAT64,                    -- Цена ₽/м² (или ₽/шт)
  unit            STRING DEFAULT 'm2',        -- m2 | sht
  cur             STRING DEFAULT 'RUB',       -- Валюта

  -- Обогащённые поля (заполняет catalog-enricher)
  sheet_kind      STRING,                     -- PROFNASTIL | METAL_TILE | ACCESSORY | SANDWICH | OTHER
  profile         STRING,                     -- С-20 | Монтеррей | ...
  thickness_mm    FLOAT64,                    -- 0.4 | 0.45 | 0.5 | 0.7
  coating         STRING,                     -- Полиэстер | Пурал | Оцинковка | ...
  color_code      STRING,                     -- RAL3005 | RR32 | Zn | DECOR:вишня
  color_system    STRING,                     -- RAL | RR | DECOR | NONE
  width_work_mm   INT64,                      -- 1100 (рабочая ширина)
  width_full_mm   INT64,                      -- 1150 (полная ширина)
  weight_kg_m2    FLOAT64,                    -- Вес кг/м² (опционально)
  
  -- Для доборных элементов и сэндвич-панелей
  sub_type        STRING,                     -- Планка карнизная | Стеновая | Кровельная
  dimensions_mm   STRING,                     -- 100×69×2000
  filler          STRING,                     -- ПИР | Минвата | ППУ (для сэндвичей)
  
  -- Служебные поля
  enrichment_status  STRING DEFAULT 'RAW',    -- RAW | PARTIAL | COMPLETE | FAILED
  enrichment_version STRING,                  -- v7.0
  enriched_at        TIMESTAMP,
  import_job_id      STRING,                  -- Из какого импорта
  created_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP(),
  updated_at         TIMESTAMP DEFAULT CURRENT_TIMESTAMP(),
  
  -- Исходные данные (для дебага и AI-контекста)
  raw_json        STRING,                     -- Вся исходная строка как JSON
)
PARTITION BY DATE(updated_at)
CLUSTER BY organization_id, sheet_kind, profile;
```

### 5.3 Приоритет источников для каждого поля

Каждое поле определяется по **цепочке приоритетов**. Если высший источник дал ответ — нижние не проверяются.

#### profile (профиль)

```
1. Regex из title/sku: (С|C|Н|H|НС|HC|МП|MP)-?\d{1,3}
2. Regex для металлочерепицы: Монтеррей|Каскад|Супермонтеррей
3. Alias из settings: bot_settings.pricing.profile_aliases
4. AI Gemini (confidence > 0.7)
5. null → enrichment_status = PARTIAL → вопрос пользователю
```

Нормализация: `С20` → `С-20`, `C-20` (лат.) → `С-20` (кир.), `МП20` → `МП-20`

#### thickness_mm (толщина)

```
1. Regex: (\d+[.,]\d+)\s*мм  или  t=(\d+[.,]\d+)  или  -(\d+[.,]\d+)-
2. Позиционный вывод из SKU-паттерна: МП-20-R-0.5-PE → 0.5
3. AI Gemini
4. null → PARTIAL → вопрос

Валидация: 0.3 ≤ x ≤ 1.5 (для кровли)
           10 ≤ x ≤ 300 (для сэндвич-панелей)
```

#### coating (покрытие)

```
1. Точное совпадение из settings: bot_settings.pricing.coatings {"МатПЭ": "Полиэстер матовый"}
2. Точное совпадение из defaults: встроенный словарь COATING_TOKENS
3. AI Gemini с enum ограничением
4. null → PARTIAL → вопрос (с suggestions от AI)

Словарь COATING_TOKENS:
  ПЭ/PE/Полиэстер/Polyester → "Полиэстер"
  МатПЭ/MatPE/Мат.ПЭ/drap  → "Полиэстер матовый"
  Пурал/Pural/PU            → "Пурал"
  ПВДФ/PVDF                 → "PVDF"
  ПВХ/PVC/Пластизол         → "Пластизол"
  Zn/оц/оцинковка           → "Оцинковка"
  Printech/принтек           → "Printech"
  Ecosteel/экостил           → "Ecosteel"
```

#### color_code (цвет)

```
1. RAL regex: RAL\s*(\d{4}) → проверить в dim_ral_colors → "RAL3005"
2. Голый 4-значный код: \b(\d{4})\b → проверить в dim_ral_colors whitelist
3. RR regex: RR\s*(\d{2}) → "RR32"
4. Alias из settings: bot_settings.pricing.colors.ral_aliases {"зел.мох": "RAL6005"}
5. Alias из settings: bot_settings.pricing.colors.decor_aliases {"вишня": {"kind":"RAL","ral":"RAL3005"}}
6. AI Gemini
7. Особый случай: coating = "Оцинковка" → color_code = "Zn", color_system = "NONE"
8. null → PARTIAL → вопрос
```

#### width_work_mm, width_full_mm (ширины)

```
1. Lookup по profile из dim_profiles
2. Из title если явно указано: \d{3,4}\s*[×x]\s*(\d{3,4})
3. Из settings: bot_settings.pricing.widths_selected

ПРАВИЛО: width_full_mm ВСЕГДА из dim_profiles (не из прайса)
         width_work_mm — приоритет из прайса, fallback на dim_profiles
```

### 5.4 Классификация по категориям

```
sheet_kind определяется по приоритетам:

1. ACCESSORY (защита — высший приоритет):
   regex: планка|конёк|отлив|ендова|саморез|водосток|снегозадержатель|уголок
   
2. SANDWICH:
   regex: сэндвич|панел[ьи].*стен|панел[ьи].*кровл
   
3. PROFNASTIL:
   regex: ^(С|C|Н|H|НС|HC|МП|MP)-?\d{1,3}$ (профиль)
   ИЛИ keyword: профнастил
   
4. METAL_TILE:
   keyword: металлочерепица|монтеррей|каскад|супермонтеррей
   
5. OTHER:
   всё что не попало выше
```

### 5.5 API-эндпоинты catalog-enricher

#### POST /api/enrich/dry_run

**Что делает**: анализ без изменений в BQ. Возвращает превью и вопросы.

```
Запрос:
{
  organization_id: "uuid",
  import_job_id: "uuid" | "current",
  scope: { only_where_null: true, limit: 5000 },
  ai_suggest: false              // true = медленнее, но AI заполнит больше
}

Ответ:
{
  ok: true,
  run_id: "run_abc123",          // для последующего apply
  profile_hash: "sha256...",     // хеш состояния dim_profiles
  stats: {
    rows_scanned: 71316,
    by_kind: {
      PROFNASTIL: 42000,
      METAL_TILE: 15000,
      ACCESSORY: 8000,
      SANDWICH: 2000,
      OTHER: 4316
    },
    complete: 63200,             // enrichment_status = COMPLETE
    partial: 5300,               // PARTIAL — есть пробелы
    failed: 0
  },
  patches_sample: [ ... ],      // 50-100 примеров обогащённых строк
  questions: [ ... ]             // до 20 паттерных вопросов
}
```

#### POST /api/enrich/preview_rows

**Что делает**: возвращает строки из BQ с фильтрацией для UI.

```
Запрос:

  organization_id: "uuid",
  filter: {
    sheet_kind: "PROFNASTIL",
    profile: "С-20",
    thickness_mm: 0.5,
    coating: null,                // null = показать строки без покрытия
    enrichment_status: "PARTIAL"
  },
  q: "поиск по title",
  limit: 50,
  offset: 0,
  sort: "title"
}

Ответ:
{
  ok: true,
  rows: [ ... ],
  total: 340,
  has_next: true
}
```

## 7. AI-пайплайн: как ИИ работает с прайсом

### 7.1 Двухуровневая архитектура

```
Уровень 1: ДЕТЕРМИНИСТИЧЕСКИЙ (regex + token matching)
  → Обрабатывает 90%+ строк
  → Мгновенный, предсказуемый
  → Использует: regex, словари, dim_profiles, settings

Уровень 2: AI (Vertex AI Gemini)
  → Обрабатывает ~8-10% строк (PARTIAL после Уровня 1)
  → 3-5 сек на batch
  → Модель: gemini-2.5-flash через Vertex AI SDK
  → Аутентификация: Service Account (Application Default Credentials)
  → НЕ через Lovable AI Gateway (сервис внутри GCP → прямой доступ)
```

### 7.2 Когда вызывается AI

AI вызывается ТОЛЬКО в двух случаях:

**Случай A: dry_run с ai_suggest=true**
— AI пытается заполнить PARTIAL-строки
— Группирует строки по типу пробела (нет покрытия, нет цвета)
— Отправляет batches по 50 строк
— Возвращает patches с confidence

**Случай B: /api/enrich/chat**
— Пользователь отправляет сообщение в AI-чат
— AI получает контекст (текущий фильтр, примеры строк)
— AI возвращает structured patches через tool_use

### 7.3 Промпт для AI (Уровень 2)

```
Ты — эксперт по нормализации промышленных каталогов металлопроката.

ЗАДАЧА: Определи недостающие атрибуты товара из его названия.

ПРАВИЛА:
1. Отвечай ТОЛЬКО через tool_call. Никакого текста.
2. Если не уверен >70% → верни null для этого поля.
3. Никогда не выдумывай значения.
4. thickness_mm: число от 0.3 до 1.5
5. coating: строго из списка [Полиэстер, Полиэстер матовый, Пурал, PVDF, Пластизол, Оцинковка]
6. color_code: RAL#### или RR## или Zn
7. profile: стандартный (С-20, НС-35, Монтеррей и т.д.)

КОНТЕКСТ ОРГАНИЗАЦИИ:
{coatings_from_settings}     — уже подтверждённые алиасы покрытий
{color_aliases_from_settings} — уже подтверждённые алиасы цветов
```

### 7.4 Генерация вопросов

При dry_run система анализирует PARTIAL-строки и генерирует **паттерные вопросы** — не по отдельным строкам, а по группам с одинаковым неизвестным токеном.

```
Пример: 47 строк содержат "МатПЭ" в title, поле coating=null

Вопрос:
{
  type: "coating",
  token: "МатПЭ",
  affected_count: 47,
  examples: [
    "С-20 0.5 МатПЭ RAL3005",
    "С-21 0.45 МатПЭ RAL8017",
    "НС-35 0.7 МатПЭ RAL6005"
  ],
  suggestions: ["Полиэстер матовый", "Полиэстер"],  // от AI или heuristic
  confidence: 0.85
}

Один ответ пользователя → исправляет ВСЕ 47 строк.
```

**Приоритет вопросов**: сортировка по affected_count DESC. Максимум 20 вопросов за dry_run.

## 9. AI-чат: как работает

### 9.1 Концепция

AI-чат — **командный инструмент**. Пользователь отдаёт команды на естественном языке, AI возвращает конкретные действия с превью.

### 9.2 Расположение

Внизу центральной области. Может быть свёрнут/развёрнут. Контекстно-зависим — знает выбранный кластер.

### 9.3 Типы команд

| Команда | Пример | Действие |
|---------|--------|----------|
| Назначить покрытие | «MatPE — это матовый полиэстер» | patch coating для N строк |
| Назначить цвет | «зел.мох = RAL6005» | patch color_code |
| Установить ширину | «С-20: рабочая 1100, полная 1150» | patch widths |
| Установить толщину | «Все С-20 без толщины — 0.45» | patch thickness_mm |
| Статистика | «Сколько без покрытия?» | INFO ответ |
| Поиск | «Покажи все с "вишня"» | фильтрация таблицы |
| Исправление | «Нет, это не 6002, а 6005» | откат + переназначение |
| Подтверждение | «Да» / «Применить» | confirm pending patch |
| Отмена | «Нет» / «Отменить» | reject pending patch |

### 9.4 Правила AI-чата

1. **Всегда превью** — любое изменение показывается ДО применения
2. **Точные числа** — «затронет 47 товаров» (не «несколько»)
3. **Примеры** — минимум 3 строки Before → After
4. **Одно действие** — не смешивать покрытие и цвет в одном патче
5. **Запоминание** — каждое решение сохраняется в settings для будущих импортов
6. **Не угадывать** — если confidence < 70%, спросить
7. **Язык пользователя** — «RAL6005 — зелёный мох», не «sRGB value #5E2028»

### 9.5 Поток данных в чате

```
Пользователь печатает: "МатПЭ это матовый полиэстер"
                              │
                              ▼
Frontend → POST /api/enrich/chat (через Edge Function)
                              │
                              ▼
catalog-enricher:
  1. Распознать намерение: PATCH_COATING
  2. Найти все строки с "МатПЭ" в title → 47 шт
  3. Сформировать patches
  4. Сформировать settings_patch
  5. Вернуть preview (3 примера Before/After)
                              │
                              ▼
Frontend:
  1. Показать карточку: "47 товаров → Полиэстер матовый"
  2. Кнопки [Применить] [Отклонить]
                              │
                              ▼
Пользователь нажал [Применить]:
  1. Вызвать settings-merge (сохранить правило)
  2. Вызвать bulk_apply/start (применить в BQ)
  3. Обновить кластерное дерево (47 строк стали COMPLETE)
```

## 11. Edge Cases и ошибки

### 11.1 Конфликтные ситуации

| Ситуация | Решение |
|----------|---------|
| В title два 4-значных числа (1150 и 3005) | 1150 → ширина (>900), 3005 → RAL (в whitelist) |
| Толщина и цена рядом: «0.5 650» | Толщина ≤ 1.5, цена > 10 — контекстное разделение |
| Профиль похож на покрытие: «ПЭ-01-9003» | COATING_GUARD: если содержит ПЭ/PE/PU → это покрытие, не профиль |
| Один SKU = несколько толщин | Ошибка данных → пометить, спросить пользователя |
| Цена = 0 | Допустимо (бесплатные образцы), не ошибка |
| Длинный title (>200 символов) | Обрезать для AI, сохранить полный в raw_json |

### 11.2 Таймауты и большие данные

| Проблема | Решение |
|----------|---------|
| dry_run > 55 сек (Edge Fn limit) | Sample mode: предложить пользователю retry с limit=250/500 |
| apply > 5 мин | Async обязательно. Polling каждые 3 сек. Max 15 мин. |
| > 100k строк | Пагинация в preview_rows. Dry_run по sample. |
| AI rate limit (Vertex) | Retry с exponential backoff. Max 3 retries. |

### 11.3 Мismatch при повторном импорте

```
Ситуация: пользователь загрузил прайс, нормализовал, через месяц загружает обновлённый.

Решение:
1. Новые строки → полный пайплайн (regex + AI)
2. Существующие строки (совпадение по bq_id) → сравнить цену
3. Если цена изменилась → обновить price_rub_m2, сохранить атрибуты
4. Если строка пропала → НЕ удалять, пометить is_active=false
5. Сохранённые правила (settings) → применяются автоматически
   → повторный импорт от того же поставщика = 97%+ автоматизация
```
